# Состояние REST API фреймворков в 2025–2026 и рекомендации для iohttp

**Ландшафт HTTP-фреймворков в 2025–2026 годах радикально изменился: типобезопасность, автогенерация OpenAPI и встроенная наблюдаемость стали базовыми ожиданиями, а не конкурентными преимуществами.** C/C++ экосистема критически отстаёт — ни один существующий фреймворк не поддерживает HTTP/3, лишь один (oatpp) генерирует OpenAPI, и ни один не предоставляет встроенную валидацию запросов. Проект iohttp на базе C23, io_uring и wolfSSL имеет уникальную возможность занять пустую нишу: **высокопроизводительный C-сервер с developer experience уровня FastAPI/Huma**. Ключевые технологии — инкрементальное потребление буферов io_uring (ядро 6.12+), kTLS для нулевого копирования HTTPS и yyjson для сериализации — позволяют достичь производительности, превышающей Drogon, при значительно лучшем DX.

## Лидеры по экосистемам: кто задаёт стандарты

### Go: зрелость и прагматизм

**Gin** (v1.12.0, 81K+ звёзд) остаётся самым популярным Go-фреймворком с radix-tree маршрутизатором и нулевыми аллокациями, но не имеет встроенной поддержки OpenAPI. **Echo** (v5.0, стабилизация до марта 2026) предлагает встроенный rate limiter, автоматический TLS через Let's Encrypt и централизованную обработку ошибок. **Fiber** (v3.0, на базе Fasthttp) показал **338 096 RPS** в тесте Fortunes TechEmpower Round 23 — второй результат среди популярных фреймворков после ASP.NET.

Главное открытие Go-экосистемы — **Huma** (v2.34+), фактический «FastAPI для Go». Huma генерирует OpenAPI 3.1 из кода, выполняет валидацию по JSON Schema 2020-12, поддерживает CBOR, условные запросы (If-Match/If-Unmodified-Since) и автоматический PATCH (JSON Merge Patch). Он router-agnostic: работает поверх Chi, Gin, Fiber или стандартного `http.ServeMux`. **ConnectRPC** (v1.17+) решает задачу унификации протоколов — один обработчик поддерживает Connect, gRPC и gRPC-Web с curl-совместимым JSON API.

### Rust: производительность и безопасность типов

**Axum** (v0.8.8, Tokio team) стал самым популярным Rust-фреймворком к 2025 году, обогнав Actix-web в опросах разработчиков. Его ключевое преимущество — интеграция с Tower middleware и 100% safe Rust (`#![forbid(unsafe_code)]`). В v0.8 синтаксис путей изменён на OpenAPI-совместимый `/{param}`. **Actix-web** (v4.12.1) сохраняет лидерство по производительности — **320 144 RPS** в Fortunes (Round 23) и ~10–15% преимущество над Axum под высокой нагрузкой.

**Poem** (v3.0) выделяется лучшей в Rust интеграцией с OpenAPI через процедурные макросы: `#[derive(Object)]`, `#[derive(ApiResponse)]`, `#[derive(SecurityScheme)]` генерируют спецификацию на этапе компиляции. **Salvo** (v0.84+) — **единственный фреймворк с продакшн-поддержкой HTTP/1+2+3** через Quinn/QUIC, с встроенным ACME/Let's Encrypt и WebTransport.

### Java/Kotlin: виртуальные потоки и нативная компиляция

**Spring Boot 3.5** (финальный 3.x минор, Spring Boot 4.0 планируется на ноябрь 2025) интегрировал виртуальные потоки Java 21+ (`spring.threads.virtual.enabled=true`), нативные образы GraalVM и структурированное логирование. Поддержка RFC 7807 ProblemDetails — встроенная. **Quarkus** (v3.28.2) обеспечивает старт менее 10 мс в нативном режиме, ~20 МБ памяти и build-time CDI без рефлексии. **Micronaut** (v4.10) уникален compile-time DI с нулевой рефлексией. **Ktor** (v3.4.0) добавил встроенный DI-модуль, сжатие zstd и генерацию OpenAPI через compiler plugin.

### Python: Pydantic v2 и типизация как движущая сила

**FastAPI** (v0.115+) остаётся золотым стандартом DX: Pydantic v2 с Rust-ядром обеспечивает ~10x ускорение валидации, автоматическая генерация OpenAPI 3.1 из type hints, `Depends()` для DI. **Litestar** (v2.15) предлагает DTO-слой с codegen-бэкендом и поддержку **нескольких библиотек валидации одновременно** (Pydantic, msgspec, dataclasses, attrs). msgspec — дефолтный сериализатор — быстрее Pydantic v2. Встроенная интеграция с OpenTelemetry — уникальная для Python-фреймворков.

### TypeScript/Node: мультирантайм и end-to-end типизация

**Hono** (v4.12) работает на **8+ рантаймах** (CF Workers, Deno, Bun, Node, Lambda, Fastly, Vercel, Netlify) с нулевыми зависимостями и весом ~14 КБ. RegExpRouter компилирует все маршруты в единое регулярное выражение. **Elysia** (v1.4) на Bun достигает **397–533K RPS** (plaintext) благодаря AOT-валидации TypeBox (~18x быстрее Zod) и статическому анализу кода. **Fastify** (v5.5) уникален JSON Schema как first-class citizen: схема одновременно валидирует запросы и ускоряет сериализацию ответов через `fast-json-stringify`.

### Другие экосистемы

**Phoenix** (Elixir, v1.8.5) — непревзойдённая отказоустойчивость BEAM VM с миллионами одновременных соединений и LiveView для real-time UI без JavaScript. **Vapor** (Swift, v4.100+) обеспечивает общую кодовую базу iOS↔сервер; Vapor 5 планирует первоклассную поддержку OpenAPI и HTTP/3.

## Ожидаемые возможности REST API в 2026 году

### OpenAPI 3.2 и потоковые ответы стали стандартом

**OpenAPI 3.2.0** (сентябрь 2025) формализовал поддержку **SSE, JSON Lines и MCP** через `itemSchema` и `prefixEncoding`. SSE стал де-факто стандартом потоковой передачи для LLM — OpenAI, Anthropic, Google используют `text/event-stream` для покоеновой трансляции. Model Context Protocol (MCP, Anthropic) использует SSE как транспорт для коммуникации AI-агентов и **мандатирует OAuth 2.1** для авторизации. Также добавлены: структурированные теги с иерархией, метод QUERY через `additionalOperations`, улучшенная поддержка multipart и device authorization flow для OAuth.

Тренд code-first vs spec-first конвергирует: FastAPI лидирует в code-first, но enterprise всё чаще использует оба подхода. Huma, Poem и Elysia демонстрируют, что генерация OpenAPI может быть побочным эффектом типобезопасного кода без дополнительных усилий.

### Сжатие: zstd выходит на сцену

**Zstandard** (zstd) активно набирает долю с марта 2024 (Chrome 123). Поддерживается всеми основными браузерами кроме Safari, используется Cloudflare по умолчанию. Ключевое преимущество: **42% быстрее Brotli** при сопоставимых коэффициентах сжатия. Для динамических API-ответов zstd — оптимальный выбор. Фреймворк должен поддерживать все три алгоритма (gzip, brotli, zstd) через `Accept-Encoding`.

### Курсорная пагинация и RFC 9457 Problem Details

**Курсорная пагинация** (keyset) настоятельно рекомендуется для больших и динамических наборов данных — Slack, Stripe, Facebook, Twitter используют её. **RFC 9457** (Problem Details, июль 2023) заменил RFC 7807 и должен быть форматом ошибок по умолчанию: поля `type`, `title`, `status`, `detail`, `instance` плюс поддержка множественных проблем и реестр типов. ASP.NET Core, Quarkus и Spring Boot уже интегрировали его.

**OAuth 2.1** (draft-15, март 2026) фактически принят: PKCE обязателен для всех потоков, implicit и ROPC удалены, refresh-токены требуют ротации. Идемпотентность формализуется в **draft-ietf-httpapi-idempotency-key-header-07** (октябрь 2025). **RateLimit headers** (draft-10, сентябрь 2025) стандартизируют формат `RateLimit-Policy` и `RateLimit` через Structured Fields.

## Критические пробелы C/C++ экосистемы

### Ни одного полноценного HTTP/3 фреймворка

Зрелые QUIC-библиотеки на C/C++ существуют — **ngtcp2** (4.94 Gbps, насыщает 10G с jumbo frames), **MsQuic** (10K+ звёзд), **lsquic**, **mvfst** (Meta), **quiche** (Cloudflare) — но **ни один C/C++ веб-фреймворк их не интегрировал**. Drogon частично поддерживает HTTP/2; Crow, oatpp и Pistache — только HTTP/1.1. В Rust Salvo уже предлагает встроенный HTTP/1+2+3 через Quinn.

### Продуктивность вдвое ниже Go и Rust

Внутреннее исследование Google (представлено на Rust Nation UK 2024) показало: **команды C++ тратят вдвое больше времени** на реализацию идентичных сервисов по сравнению с Go или Rust. Ключевые факторы: компиляция (минуты vs секунды), отсутствие hot reload, сложность CMake+vcpkg/Conan, ручное управление памятью, криптические ошибки шаблонов, фрагментация экосистемы и отсутствие стандартной HTTP-библиотеки.

### Что разработчикам приходится писать вручную

В C/C++ фреймворках нет: валидации запросов по схеме (только oatpp DTO), JWT/OAuth, rate limiting, CORS (только oatpp макрос), RFC 9457 Problem Details, Kubernetes health checks, graceful shutdown, OpenTelemetry, dependency injection, миграций БД. Минимальный REST-endpoint на Drogon требует header-файл, implementation-файл, объявления макросов и изменений CMakeLists — эквивалент FastAPI занимает 3 строки.

### Что сделал бы «FastAPI для C»

Oatpp ближе всего — его DTO-система + Swagger-интеграция покрывают часть задач. Полноценный аналог требует: (1) C23/C26 рефлексию для автосериализации без макросов, (2) валидацию на уровне Pydantic, (3) однофайловую/модульную инклюзию, (4) корутинный async с простым рантаймом, (5) constexpr-генерацию OpenAPI, (6) горячую перезагрузку через shared library. Реалистичная оценка: C никогда не сравняется с FastAPI по DX, но комбинация C23 + кодогенерация + макросы может значительно сократить разрыв.

## io_uring: архитектура HTTP-сервера нового поколения

### Революционные возможности ядер 6.10–6.12

**Ядро 6.10** принесло **send/recv bundles** (`IORING_RECVSEND_BUNDLE`) — одна операция использует несколько буферов из provided buffer ring. Бенчмарки Jens Axboe: для 32-байтных пакетов пропускная способность выросла с **1183 до 4853 Мбит/с** (4x). Для sends появились provided-буферы (`IORING_FEAT_SEND_BUF_SELECT`), а порог эффективности zero-copy снизился до **~3000 байт**.

**Ядро 6.11** добавило нативные `IORING_OP_BIND` и `IORING_OP_LISTEN` — критичные для HTTP-серверов, использующих direct file descriptors через `IORING_OP_ACCEPT`.

**Ядро 6.12** — наиболее значимое обновление: **инкрементальное потребление буферов** (`IOU_PBUF_RING_INC`) позволяет 32 КБ буферу обслуживать множественные recv-операции, каждая начинающая с того места, где остановилась предыдущая. Индикатор: флаг `IORING_CQE_F_BUF_MORE`. Также добавлены: двухуровневые таймауты (min-timeout для адаптивного батчинга), абсолютные таймауты с выбором clock source, клонирование буферов между кольцами.

**Ядро 6.13** позволяет динамически изменять размер колец, добавляет гибридный I/O polling и частичное клонирование буферных таблиц.

### Оптимальная конфигурация для HTTP-сервера

Рекомендуемая архитектура — **thread-per-core** с одним io_uring кольцом на поток:

```
IORING_SETUP_DEFER_TASKRUN + SINGLE_ISSUER + COOP_TASKRUN + TASKRUN_FLAG
SQ = 256, CQ = 1024 (4x для запаса multishot overflow)
```

**DEFER_TASKRUN** (ядро 6.1+) — лучший выбор для однопоточных серверов: task_work выполняется **только** при `io_uring_enter()` с `GETEVENTS`, обеспечивая полный контроль над батчингом и устраняя нежелательные прерывания. **SQPOLL** оправдан лишь при выделении отдельного ядра CPU и требовании минимальной латентности. Исследование TU Darmstadt (2025) показало: **DEFER_TASKRUN + NAPI busy polling даёт лучшую общую латентность**, опережая SQPOLL.

### Стратегия буферов: три группы

Стратегия provided buffer rings для HTTP: **Группа 0** (4 КБ, 1024 буфера) для заголовков запросов; **Группа 1** (16–32 КБ, 256 буферов) для тел запросов/ответов с IOU_PBUF_RING_INC; **Группа 2** (64+ КБ, 64 буфера) для загрузки файлов. Буферы recycling обратно через `io_uring_buf_ring_add()` + `io_uring_buf_ring_advance()` сразу после обработки.

### TLS через kTLS: нулевой syscall на запрос

Архитектура: TLS-рукопожатие в userspace (wolfSSL) → установка kTLS через `setsockopt(SOL_TLS, TLS_TX/RX, crypto_info)` → io_uring send/recv прозрачно шифруют/дешифруют в ядре. Бонус: `sendfile()` и `splice()` работают через kTLS — статические файлы обслуживаются без копирования в userspace. Проект **tarweb** (2025, Rust + io_uring + kTLS) продемонстрировал HTTPS без per-request syscalls. Проект **Tempesta** работает над полным TLS-рукопожатием в ядре (40–80% прогнозируемого улучшения).

### Zero-copy: пороги и бенчмарки

Данные IORING_OP_SEND_ZC с реальной сетевой картой: **\<1 КБ** — нет выгоды; **1.5 КБ** — +4.5%; **4 КБ** — +22%; **8 КБ** — +84% (CPU-bound). Данные Google/Meta: io_uring ZC достигает **90.4 Gbps** vs epoll 68.8 Gbps при MTU 1500 (+31%). Рекомендация: использовать ZC для ответов >4 КБ; флаг `IORING_SEND_ZC_REPORT_USAGE` позволяет проверить реальное использование ZC.

### io_uring vs epoll: нюансы

io_uring **не всегда быстрее**. При batch=1 он **медленнее** epoll (2600 нс vs 2200 нс с CPU-mitigation). Преимущество появляется при больших батчах и включённых Spectre/Meltdown mitigations (syscall overhead 230 нс → 700 нс). **Безопасность**: Google отключил io_uring на ChromeOS, продакшн-серверах и Android-приложениях; 60% эксплойтов bug bounty 2022 целились в io_uring. Необходимо проверять `/proc/sys/kernel/io_uring_disabled` и учитывать seccomp-профили контейнеров.

## C23: арсенал для безопасного HTTP-сервера

### Целостность данных через checked-арифметику

**`<stdckdint.h>`** (`ckd_add`, `ckd_sub`, `ckd_mul`) — критически важная функция для HTTP-серверов. Парсинг Content-Length, вычисление размеров буферов, валидация Range-запросов — все операции подвержены integer overflow. C23 заменяет ненадёжные ручные проверки типа `if (a > SIZE_MAX - b)` на типобезопасные макросы, использующие аппаратные флаги переполнения. GCC поддерживает через `__builtin_*_overflow` с версии 5; Clang 18+ предоставляет заголовок.

### Безопасная очистка памяти

**`memset_explicit`** гарантированно не оптимизируется компилятором — критично для очистки TLS-ключей, паролей, сессионных токенов. Обычный `memset` часто удаляется оптимизатором, оставляя секреты в памяти. Требует glibc 2.39+.

### Встраивание статических ресурсов

**`#embed`** позволяет встроить бинарные файлы (страницы ошибок 404/500, favicon, сертификаты) непосредственно в исходный код. GCC 15 демонстрирует: встраивание 9.2 МБ файла: подход xxd — **30+ секунд** компиляции, `#embed` — **менее 1 секунды**. GCC 15+, Clang 19+.

### Типобезопасность и атрибуты

**`[[nodiscard("check for write errors")]]`** — незаменим для I/O операций: игнорирование возвращаемого значения `send_response()` вызывает предупреждение компилятора. **`nullptr`** устраняет класс ошибок с вариадическими функциями. Перечисления с фиксированным типом (`enum http_status : uint16_t`) обеспечивают точный контроль хранения протокольных констант. **`unreachable()`** позволяет оптимизатору устранить проверки границ в конечных автоматах парсера.

### C2y: defer меняет всё

Техническая спецификация **`defer`** (TS 25755) после встречи WG14 в феврале 2025 считается готовой; GCC имеет патч. `defer { close(fd); }` выполняется при выходе из блока — революция для управления ресурсами в C-серверах, где сейчас требуется goto-based cleanup или `[[gnu::cleanup]]`. Также принят `_Countof` для безопасной итерации массивов, именованные циклы и case-диапазоны (`case 0x41 ... 0x5A:`) — идеальны для HTTP-парсеров.

### Статус компиляторов: GCC 15 — ключевой рубеж

**GCC 15** (апрель 2025) сделал C23 режимом по умолчанию (`-std=gnu23`) и «по существу полностью» поддерживает стандарт. Безопасно использовать в 2026: `[[nodiscard]]`, `nullptr`, `typeof`, `constexpr`, `#embed`, `<stdckdint.h>`, `auto`, все стандартные атрибуты. `<stdbit.h>` и `memset_explicit` зависят от версии glibc.

## Безопасность: от OWASP до request smuggling

### OWASP API Security Top 10 (2023): ключевые риски для C-сервера

Обновлений 2025 нет — редакция 2023 актуальна. Для C-фреймворка наиболее критичны: **API4 (Unrestricted Resource Consumption)** — необходимы конфигурируемые лимиты размера тела запроса, заголовков, таймаутов, соединений на IP; **API1 (BOLA)** — middleware-хуки авторизации на уровне каждого обработчика; **API8 (Security Misconfiguration)** — безопасные дефолты: минимум включённых функций, рестриктивный CORS, security-заголовки по умолчанию.

### Память: C23 + инструменты = приемлемая безопасность

Обязательный CI-пайплайн: **ASan + UBSan** для всех тестов, **Valgrind** для интеграционных тестов, **AFL++/libFuzzer** с ASan для фаззинга каждого парсера (HTTP request line, headers, chunked encoding, URL decoding, multipart, query string). Production-флаги: `-fstack-protector-strong -D_FORTIFY_SOURCE=2 -fPIE -pie -Wl,-z,relro,-z,now -fstack-clash-protection -fcf-protection`. Arena-аллокатор per-request устраняет use-after-free и утечки: вся память запроса освобождается одной операцией.

OpenSSF Memory Safety Continuum (апрель 2025) предлагает практический фреймворк инкрементального улучшения безопасности памяти даже в C-кодовых базах. NSA и CISA (июнь 2025) рекомендуют memory-safe языки для новых проектов, но признают необходимость C для системного ПО.

### Request smuggling: имплементация с нуля устраняет проблему

Исследование James Kettle (DEF CON 33, 2025): «HTTP/1.1 должен умереть» — smuggling — inherent flaw протокола. Для нового сервера **критические правила**: отклонять запросы с одновременными Content-Length и Transfer-Encoding; строго парсить заголовки (нет CRLF в значениях, нет null bytes, нет пробелов в методе); закрывать соединение при любой ошибке парсинга; защита от Rapid Reset (CVE-2023-44487) — лимит на rate reset'ов на соединение; HPACK bomb — лимит размера таблицы и максимум декодированного заголовка.

### wolfSSL: специфические рекомендации

Последние CVE — CVE-2025-11936 и CVE-2025-11933 (duplicate KeyShare DoS в TLS 1.3 ClientHello) — исправлены в wolfSSL 5.8.4. Обязательна валидация расширений ClientHello. Приоритет шифронаборов: AES-256-GCM (Intel/AMD с AES-NI), ChaCha20-Poly1305 (ARM). 0-RTT — только для идемпотентных операций (GET, HEAD, OPTIONS), никогда для POST/PUT/DELETE.

## Производительность: что делает серверы быстрыми

### TechEmpower Round 23: C/C++ и Rust лидируют

Round 23 (февраль 2025) на новом оборудовании (Xeon Gold 6330, 56 ядер, 40 Gbps) показал 3–4x улучшение. **Drogon** (C++) стабильно входит в топ-5 по JSON/plaintext. 6 из 10 лидеров — Rust-фреймворки. ASP.NET Core лидирует в Fortunes (**609 966 RPS**) благодаря агрессивным оптимизациям .NET 9.

### JSON: yyjson — оптимальный выбор для C

Бенчмарки на twitter.json: **yyjson** — 2.39 GB/s парсинг, **2.01 GB/s сериализация** — быстрейшая C-библиотека для обоих направлений. simdjson быстрее для парсинга с On Demand API (до 7 GB/s при известных полях), но сериализация — лишь 0.80 GB/s. yyjson написан на ANSI C (C89), портабелен, идеально совместим с arena-аллокаторами. cJSON в **5x медленнее**, Jansson — в **25x**.

### HTTP-парсинг: SIMD реально помогает

**picohttpparser** (C, парсер H2O) использует SSE 4.2 PCMPESTRI для SIMD-сканирования — **68–90% быстрее** non-SIMD версии. CloudFlare заменила SSE 4.2 на AVX2: обработка 32 байт за шаг с развёрткой до 128 байт — **1.78–2x ускорение** над PCMPESTRI. Для HTTP/1.1 SIMD-парсинг значим; для HTTP/2 и /3 бинарное фреймирование снижает актуальность.

### QUIC-библиотеки: ngtcp2 лидирует

На 10 Gbps линке (MTU 1500): **ngtcp2** (C) — 4.94 Gbps; **quiche** (Rust) — 4.16 Gbps; **lsquic** (C) — 3.6–3.88 Gbps. С jumbo frames (MTU 9000): ngtcp2 **насыщает 10G** (9.97 Gbps). TCP/TLS на том же линке — ~8 Gbps. QUIC уже конкурентоспособен по пропускной способности; HTTP/3 снижает TTFB на **12.4%** (Cloudflare), задержку на мобильных устройствах — на **30%** (Catchpoint/Akamai), а при потерях пакетов — до **88%**.

### Аллокаторы: mimalloc для HTTP-серверов

**mimalloc** (Microsoft, v3.1.5) превосходит все ведущие аллокаторы в комплексных бенчмарках: лучшая скорость мелких аллокаций, консистентная производительность. **tcmalloc** лидирует для аллокаций >1 КБ. **jemalloc** — лучшая латентность и минимальная фрагментация для долгоживущих процессов, но проект считается «полумёртвым» в 2025. Быстрейшие фреймворки обходят аллокаторы полностью: arena-аллокаторы, предвыделенные пулы буферов, стековые аллокации в hot paths.

## Рекомендации для проекта iohttp

### Обязательные возможности (MVP)

1. **HTTP/1.1 + HTTP/2 + HTTP/3** — полная поддержка RFC 9112/9113/9114. Использовать nghttp2 для HTTP/2, ngtcp2+nghttp3 для HTTP/3. Это немедленно выделит iohttp среди **всех** C/C++ фреймворков.

2. **Radix-trie маршрутизатор** с синтаксисом `/{param}` (совместим с OpenAPI) и wildcards `/{*path}`. Нулевые аллокации, приоритет: статические > параметризованные > wildcard.

3. **Middleware pipeline** — цепочка `handler(ctx, next)` с поддержкой per-route, per-group и global middleware. Встроенные: CORS, compression (gzip+brotli+zstd), request ID, security headers, body limit, timeout.

4. **RFC 9457 Problem Details** — дефолтный формат ошибок. Встроенные функции `iohttp_error(ctx, 404, "Not Found", "/errors/not-found")`.

5. **Структурированное логирование** в JSON с W3C Trace Context (traceparent/tracestate). Endpoint `/metrics` в формате Prometheus.

6. **Kubernetes probes** — встроенные `/health/live`, `/health/ready`, `/health/startup` с настраиваемыми проверками.

7. **Graceful shutdown** — обработка SIGTERM, drain in-flight запросов с конфигурируемым таймаутом, закрытие соединений.

8. **Rate limiting** — token bucket и sliding window с хранилищем in-memory; API для внешних хранилищ (Redis). Заголовки по draft-ietf-httpapi-ratelimit-headers-10.

9. **JWT-валидация** с обязательной фиксацией алгоритма (предотвращение algorithm confusion), проверкой exp/iss/aud, поддержкой JWKS-ротации.

10. **TLS 1.3** через wolfSSL с kTLS для data-path. ACME/Let's Encrypt автоматизация. Минимальный набор: AES-256-GCM, AES-128-GCM, ChaCha20-Poly1305.

11. **Arena-аллокатор per-request** — bump allocator с linked list страниц. Все аллокации запроса из arena; освобождение одной операцией.

12. **JSON через yyjson** — самая быстрая C-библиотека для парсинга и сериализации.

### Рекомендуемые возможности (дифференциация)

1. **Декларативная валидация запросов** через макросы-описатели полей с типами, min/max, pattern, required — по образцу oatpp DTO, но с C23 typeof/constexpr для типобезопасности. Генерация JSON Schema из описателей.

2. **Автогенерация OpenAPI 3.1** из маршрутов и описателей — основная функция, отсутствующая почти во всех C-фреймворках. Встроенный Swagger UI через `#embed`.

3. **SSE-поддержка** с типизированными событиями — критично для AI/LLM-интеграций и MCP.

4. **WebSocket** с поддержкой per-message deflate.

5. **ETag/условные запросы** — автоматическая генерация ETag из хеша ответа, поддержка If-None-Match (304) и If-Match (412).

6. **Idempotency-Key** по draft-07 — хранение результатов с TTL, возврат кешированного ответа, 409 при конкурентных запросах.

7. **mTLS** с валидацией цепочки сертификатов и поддержкой RFC 9440 Client-Cert для прокси-сценариев.

8. **Content negotiation** — JSON (дефолт), MessagePack, CBOR через Accept header.

9. **HTTP Message Signatures** (RFC 9421) — для финансовых и B2B API.

10. **Zero-copy send** (IORING_OP_SEND_ZC) для ответов >4 КБ с автоматическим fallback для мелких ответов.

11. **Фаззинг-харнесы** в комплекте — libFuzzer-совместимые функции для каждого парсера, запускаемые в CI.

### Что НЕ стоит реализовывать

1. **ORM/миграции БД** — не задача HTTP-фреймворка на C. Предоставить асинхронный интерфейс для подключения библиотек (libpq-async, mysql-client), но не строить ORM.

2. **Template engine** — серверный рендеринг HTML не является целевым сценарием для C-фреймворка в 2026. Фокус — JSON API.

3. **GraphQL runtime** — слишком сложен для C, лучше интегрировать через gateway.

4. **Полный OAuth 2.1 authorization server** — это отдельный продукт (Keycloak, Hydra). Поддержать валидацию токенов, но не выдачу.

5. **HATEOAS/HAL полная реализация** — практическое применение ограничено; достаточно поддержки Link-заголовков для пагинации.

6. **Hot reload** — фундаментально несовместим с компилируемым C; попытки через shared library слишком хрупки.

7. **Dependency injection контейнер** — over-engineering для C. Достаточно передачи контекста через `ctx->app_data` (паттерн Axum State\<T\>).

8. **SQPOLL по умолчанию** — потребляет выделенное ядро CPU даже в idle; DEFER_TASKRUN лучше для большинства сценариев.

### Ключевые технические решения

- **Минимальная версия ядра: 6.11** (для native bind/listen с direct fd). Оптимальная: **6.12** (инкрементальное потребление буферов, min-timeout).
- **Компилятор: GCC 15+** (C23 по умолчанию). Использовать `[[nodiscard]]`, `nullptr`, `constexpr`, `<stdckdint.h>`, `#embed`, `unreachable()`, `memset_explicit`, enum с фиксированным типом.
- **JSON: yyjson** (2x+ быстрее simdjson по сериализации, чистый C, arena-совместимый).
- **HTTP-парсер: собственный** с SIMD (AVX2/SSE 4.2) по образцу picohttpparser/CloudFlare — контроль над безопасностью и интеграцией с io_uring provided buffers.
- **Аллокатор: mimalloc** как системный + arena per-request.
- **QUIC: ngtcp2 + nghttp3** — лучшая производительность среди C-библиотек, автор nghttp2 (Tatsuhiro Tsujikawa), активная разработка.
- **Архитектура: thread-per-core** — один io_uring ring на поток, DEFER_TASKRUN + SINGLE_ISSUER, shared-nothing для data structures.

## Заключение: стратегическая позиция iohttp

Рынок C/C++ HTTP-фреймворков в 2026 году — это парадокс: C/C++ **доминируют** в бенчмарках TechEmpower, но **проигрывают** по каждому аспекту developer experience. Ни один C-фреймворк не предлагает одновременно HTTP/3, OpenAPI-генерацию, валидацию запросов, встроенную наблюдаемость и современные паттерны безопасности. iohttp имеет возможность стать **первым C-фреймворком, закрывающим этот разрыв**.

Три фактора делают этот момент оптимальным: C23 (GCC 15 по умолчанию) предоставляет `[[nodiscard]]`, `<stdckdint.h>`, `#embed` и `memset_explicit` — инструменты, которых не было ещё два года назад; io_uring с ядрами 6.10–6.12 наконец дозрел до продакшн-уровня для HTTP с bundles, инкрементальным потреблением буферов и kTLS; а ngtcp2 показывает, что HTTP/3 на C может насыщать 10G линк. Если iohttp реализует Фазы 1–3 с качеством, сопоставимым с Drogon по производительности и Huma по DX, он займёт уникальную позицию в экосистеме — единственный C-фреймворк, для которого разработчикам не нужно будет «дописывать всё остальное вручную».
