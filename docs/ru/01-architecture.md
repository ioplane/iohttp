# Архитектура iohttp

## Обзор

iohttp -- это встраиваемая HTTP-серверная библиотека промышленного уровня на C23, построенная
как модульная композиция проверенных протокольных библиотек поверх Linux io_uring и wolfSSL.

**Философия проектирования:** Владеть движком ввода-вывода и связующим кодом; делегировать
разбор протоколов проверенным библиотекам. Написать ~14-23K строк интеграционного кода, а не
80K строк реализации протоколов.

---

## Основная архитектура

```
┌─────────────────────────────────────────────────────────────┐
│                      User Application                       │
│   ioh_server_create() → ioh_route_add() → ioh_server_run()    │
├─────────────────────────────────────────────────────────────┤
│                     Public C API (ioh_*)                      │
├──────────┬──────────┬──────────┬───────────┬────────────────┤
│  Router  │Middleware│  Static  │ WebSocket │      SSE       │
│  (trie)  │  chain   │  files   │ RFC 6455  │  text/stream   │
├──────────┴──────────┴──────────┴───────────┴────────────────┤
│                    HTTP Protocol Layer                       │
│  ┌──────────────┬───────────────┬──────────────────────┐    │
│  │  HTTP/1.1    │    HTTP/2     │      HTTP/3          │    │
│  │picohttpparser│   nghttp2     │  ngtcp2 + nghttp3    │    │
│  │  (stateless) │  (callbacks)  │  (wolfSSL crypto)    │    │
│  └──────────────┴───────────────┴──────────────────────┘    │
├─────────────────────────────────────────────────────────────┤
│                    TLS Layer (wolfSSL)                       │
│  TLS 1.3 · mTLS · ALPN · SNI · Session Resumption · QUIC   │
├─────────────────────────────────────────────────────────────┤
│                  I/O Engine (io_uring)                       │
│  Multishot Accept · Provided Buffers · Zero-Copy Send       │
│  Ring-Based Timers · DEFER_TASKRUN · REGISTER_RESTRICTIONS  │
├─────────────────────────────────────────────────────────────┤
│                    Linux Kernel 6.7+                         │
└─────────────────────────────────────────────────────────────┘
```

## Декомпозиция модулей

### 1. Движок ввода-вывода (`src/core/`)

Фундамент: цикл событий io_uring, управляющий всем вводом-выводом. Один реактор (режим
разработки) или мультиреакторная архитектура с отдельным кольцом на поток (продакшен).

| Компонент | Строк | Описание |
|-----------|-------|----------|
| `ioh_loop.{h,c}` | ~500 | Настройка кольца io_uring, отправка SQE, сбор CQE, колесо таймеров |
| `ioh_server.{h,c}` | ~1055 | Жизненный цикл сервера, linked timeouts, signalfd shutdown, лимиты запросов, request ID |
| `ioh_buffer.{h,c}` | ~250 | Управление предоставляемыми буферными кольцами, пул буферов |
| `ioh_conn.{h,c}` | ~385 | Конечный автомат соединений, фазы таймаутов, состояние PROXY |
| `ioh_log.{h,c}` | ~165 | Структурированное логирование с уровнями, пользовательские sink-колбэки |

**Используемые возможности io_uring:**
- `IORING_OP_ACCEPT` с `IORING_ACCEPT_MULTISHOT` -- один SQE принимает все соединения
- `IOSQE_CQE_SKIP_SUCCESS` -- подавление CQE при успешном multishot accept (снижает нагрузку на CQ)
- `IORING_OP_RECV` с предоставляемыми буферными кольцами -- ядро выбирает буфер из пула
- `IORING_OP_SEND_ZC` -- zero-copy отправка для данных > 2 KiB (ядро 6.0+)
- `IORING_OP_LINK_TIMEOUT` -- связанные таймауты для дедлайнов keepalive/заголовков/тела
- `IORING_OP_SPLICE` / `IORING_OP_SENDFILE` -- zero-copy раздача статических файлов
- `IORING_REGISTER_BUFFERS` -- закреплённая память для ускорения DMA при recv/send
- `IORING_REGISTER_FILES` -- зарегистрированные файловые дескрипторы, без поиска в таблице fd на каждую операцию
- `io_uring_setup_buf_ring` -- предоставляемое буферное кольцо для автоматического выделения буферов recv
- `IORING_SETUP_SQPOLL` -- опциональный опрос SQ на стороне ядра (НЕ по умолчанию; предпочтителен DEFER_TASKRUN)
- `IORING_REGISTER_RESTRICTIONS` -- белый список опкодов для смягчения обхода seccomp
- `IORING_OP_MSG_RING` -- передача fd/сообщений между кольцами для передачи соединений в мультиреакторной архитектуре

**Нет fallback на epoll.** io_uring обязателен. Минимальное ядро: 6.7+.

**Регистрация ограничений io_uring:** При запуске, после создания кольца и настройки буферов,
но до начала приёма соединений, сервер ОБЯЗАН вызвать `IORING_REGISTER_RESTRICTIONS` для
создания белого списка только необходимых опкодов: `RECV`, `SEND`, `SEND_ZC`, `ACCEPT`, `TIMEOUT`,
`CANCEL`, `MSG_RING`, `SPLICE`, `LINK_TIMEOUT`. Это критически важно, поскольку операции io_uring
обходят фильтры seccomp BPF -- они используют кольцо в разделяемой памяти, а не системные вызовы.
Без ограничений опкодов скомпрометированный процесс может выполнять произвольные операции
io_uring, которые seccomp не в состоянии перехватить.

**CVE-2024-0582 (use-after-free в предоставляемых буферных кольцах):** Ядра 6.4--6.6.4 содержат
уязвимость use-after-free в предоставляемых буферных кольцах. Требование минимального ядра 6.7
позволяет избежать этой CVE.

**SQPOLL vs DEFER_TASKRUN:** `IORING_SETUP_SQPOLL` является опциональным и НЕ включён по умолчанию.
Предпочтительный режим для HTTP-серверов -- `IORING_SETUP_DEFER_TASKRUN | IORING_SETUP_SINGLE_ISSUER`,
который лучше подходит для всплескового характера трафика. SQPOLL сжигает выделенное ядро CPU
для опроса кольца отправки и взаимоисключающ с `DEFER_TASKRUN`. Используйте SQPOLL для
критичных к задержкам нагрузок, где расход выделенного ядра CPU допустим.

**Эвристика динамического порога SEND_ZC:**
- Обычный асинхронный `IORING_OP_SEND` для данных < 1--2 KiB (заголовки, небольшие JSON-ответы)
- `IORING_OP_SEND_ZC` для данных > 2 KiB (большие ответы, файловые данные, потоковые тела)
- `IORING_OP_SPLICE` для статических файлов (page cache -> сокет, без отображения буферов в пространстве пользователя)
- Флаг `IORING_SEND_ZC_REPORT_USAGE` для обнаружения ситуаций, когда ядро откатывается к
  копированию (например, loopback-интерфейс), что позволяет адаптировать порог в рантайме
- Примечание: `SEND_ZC` генерирует 2 CQE на операцию -- один для завершения, один для
  уведомления о zero-copy. Цикл сбора CQE должен обрабатывать оба.

**Мультиреакторная архитектура (продакшен):**
- Одно кольцо io_uring на рабочий поток
- Каждый воркер владеет своим пулом соединений и локальностью буферов
- Стратегия слушателя: SO_REUSEPORT (на каждый воркер) или общий accept + передача
- Соединение НЕ мигрирует между кольцами при обычной работе
- Строгое владение: каждое соединение имеет ровно одного владельца-потока

**Диспетчеризация CQE:** типизированная упаковка user_data `(conn_id << 8) | op_type` для
быстрого O(1) различения операций accept/recv/send/timeout/file/tls. Без выделений в куче.

**Конечный автомат соединений:**
```
ACCEPTING → PROXY_HEADER → TLS_HANDSHAKE → PROTOCOL_NEGOTIATION
  → HTTP_ACTIVE → DRAINING → CLOSING → CLOSED
       ↓               ↓
  WEBSOCKET_ACTIVE  SSE_ACTIVE
```

### 2. Слой TLS (`src/tls/`)

Интеграция через нативный API wolfSSL (НЕ через слой совместимости с OpenSSL).

| Компонент | Строк | Описание |
|-----------|-------|----------|
| `ioh_tls.{h,c}` | ~460 | Контекст wolfSSL, пользовательские callback ввода-вывода для io_uring |
| `ioh_tls_quic.{h,c}` | планируется | QUIC-криптография через ngtcp2_crypto_wolfssl |

**Паттерн callback ввода-вывода для io_uring:**
```c
// wolfSSL читает из управляемого приложением шифробуфера
// io_uring recv → cipher_buf → wolfSSL_read() → plaintext → HTTP parser
// HTTP response → wolfSSL_write() → cipher_buf → io_uring send
//
// CRITICAL: wolfSSL_CTX_set_mode(ctx, SSL_MODE_ENABLE_PARTIAL_WRITE)
// io_uring является неблокирующим — необходимо обрабатывать WANT_READ/WANT_WRITE вручную
```

**Сериализация буферов ввода-вывода:** wolfSSL имеет единственный буфер ввода-вывода на объект
`WOLFSSL`. Это означает, что `wolfSSL_read()` и `wolfSSL_write()` на одном объекте `WOLFSSL`
ДОЛЖНЫ быть сериализованы -- параллельные вызовы повреждают внутреннее состояние. В архитектуре
с отдельным кольцом на поток и строгим владением соединениями это обеспечивается естественным
образом (один поток владеет каждым соединением, поэтому чтения и записи последовательны внутри
цикла событий).

**Обработка SIGPIPE:** `signal(SIGPIPE, SIG_IGN)` ОБЯЗАТЕЛЕН при запуске. Хотя io_uring
возвращает ошибки при записи в закрытый сокет, wolfSSL может вызвать `SIGPIPE` через свои
внутренние вызовы записи (пользовательские callback ввода-вывода в конечном счёте вызывают
запись в ядро). Без этого процесс сервера может быть убит необработанным `SIGPIPE`, когда
клиент отключается в середине ответа.

**Поддерживаемые возможности:**
- TLS 1.3 (основной), TLS 1.2 (опциональная совместимость)
- mTLS с проверкой CRL
- Согласование ALPN (h2, http/1.1)
- Callback SNI для мультитенантности
- Тикеты сессий с настраиваемым TTL
- Перезагрузка сертификатов без перезапуска
- OCSP stapling (опционально)
- QUIC-криптография для HTTP/3 через ngtcp2_crypto_wolfssl

**Примечание о лицензии wolfSSL:** Лицензионная ситуация wolfSSL требует уточнения перед
релизом. Файл `LICENSING` на GitHub указывает GPLv2, wolfssl.com/license указывает GPLv3, а
руководство wolfSSL указывает GPLv2. Если wolfSSL строго под GPLv2 (не GPLv2-или-позднее),
она несовместима с кодом GPLv3 -- эти две лицензии не могут быть объединены в одном
произведении. Рекомендация: уточнить точные условия лицензии у wolfSSL Inc. или приобрести
коммерческую лицензию перед любым публичным выпуском iohttp.

### 3. Слой HTTP-протоколов (`src/http/`)

Три реализации протоколов, разделяющие унифицированную абстракцию запроса/ответа.

| Компонент | Строк | Описание |
|-----------|-------|----------|
| `ioh_http1.{h,c}` | ~540 | Обёртка picohttpparser, объект запроса, chunked TE |
| `ioh_http2.{h,c}` | ~770 | Сессия nghttp2, мультиплексирование потоков, HPACK, управление потоком |
| `ioh_http3.{h,c}` | планируется | Транспорт QUIC через ngtcp2 + HTTP/3 через nghttp3 + QPACK + демультиплексирование CID |
| `ioh_request.{h,c}` | ~350 | Унифицированная абстракция запроса для всех протоколов |
| `ioh_response.{h,c}` | ~330 | Построитель ответов, сериализация заголовков |
| `ioh_multipart.{h,c}` | ~360 | Разбор multipart form-data (RFC 2046) |
| `ioh_proxy_proto.{h,c}` | ~370 | Декодер PROXY protocol v1/v2 |

**Унифицированный поток обработки запроса:**
```
Bytes from network
    ↓
PROXY protocol decode (if enabled on listener)
    ↓
wolfSSL decrypt (if TLS)
    ↓
ALPN → protocol-specific parser
    ↓
Unified ioh_request_t
    ↓
Router → Middleware → Handler
    ↓
ioh_response_t → protocol-specific framing → wolfSSL encrypt → io_uring send
```

**Плавная остановка HTTP/2 (двухфазный GOAWAY):**
1. Отправить GOAWAY с `last_stream_id = 2^31-1` (сигнал: новых потоков не будет)
2. Подождать не менее одного RTT для получения запросов в полёте
3. Отправить финальный GOAWAY с фактическим ID последнего обработанного потока
4. Дождаться `nghttp2_session_want_read() == 0 && nghttp2_session_want_write() == 0`
5. Закрыть соединение через io_uring

**Особенности HTTP/3 / QUIC:**
- **Демультиплексирование CID:** Multishot UDP recv доставляет смешанные пакеты от всех клиентов.
  Хеш-таблица сопоставляет Connection ID -> `ngtcp2_conn` для маршрутизации пакетов за O(1).
- **UDP GSO/GRO:** GSO объединяет несколько пакетов QUIC в один sendmsg через cmsg
  `UDP_SEGMENT`. GRO объединяет принятые пакеты. ngtcp2 поддерживает GSO нативно через
  `ngtcp2_conn_write_aggregate_pkt()`.
- **Управление перегрузкой:** Настраивается -- CUBIC (по умолчанию), NewReno, BBRv2. BBRv2
  даёт ~21% лучшую задержку p99 в сетях с потерями/мобильных сетях.
- **Миграция соединений:** ngtcp2 полностью поддерживает миграцию путей QUIC. Сервер обрабатывает
  PATH_CHALLENGE/PATH_RESPONSE и обновляет сопоставление CID-соединение.
- **`SO_REUSEPORT`** для нескольких рабочих потоков, принимающих на одном UDP-порте.
- **Плавная остановка:** `nghttp3_conn_shutdown()` -> отказ от новых потоков -> опрос
  `nghttp3_conn_is_drained()` -> закрытие QUIC-соединения после завершения всех потоков.

### 4. Маршрутизатор (`src/router/`)

| Компонент | Строк | Описание |
|-----------|-------|----------|
| `ioh_radix.{h,c}` | ~530 | Radix trie (сжатое префиксное дерево), внутренний |
| `ioh_router.{h,c}` | ~530 | Деревья по методам, авто-405/HEAD, коррекция пути |
| `ioh_route_group.{h,c}` | ~310 | Вложенные группы с композицией префиксов |
| `ioh_route_inspect.{h,c}` | ~130 | Обход маршрутов, интроспекция, привязка к liboas |
| `ioh_middleware.{h,c}` | ~190 | Цепочка middleware, обработчик ошибок, паттерн next() |

**Наследие дизайна:** Radix trie + деревья по методам (httprouter), приоритет static > param >
wildcard (httprouter/echo/bunrouter), handler возвращает ошибку (bunrouter/echo), группы
маршрутов (Express.Router), авто-405/HEAD (httprouter/FastRoute), обнаружение конфликтов
(matchit/Axum), интроспекция маршрутов (gorilla/mux), прикрепление метаданных для OpenAPI (FastAPI).

**Возможности маршрутизации:**
- **Radix trie** со сжатым разделением префиксов, отдельное дерево на HTTP-метод
- **Детерминированный приоритет** (не зависит от порядка): static > `:param` > `*wildcard`
- **Регистрация по методам**: `ioh_router_get()`, `ioh_router_post()` и т.д.
- **Параметры пути**: `/api/users/:id/config` -- типизированное извлечение (string, i64, u64, bool)
- **Маршруты с подстановкой**: `/static/*path` -- захватывает оставшуюся часть пути
- **Вложенные группы маршрутов**: композиция префиксов + наследование middleware по группам
- **Авто-405 Method Not Allowed**: путь совпадает с другим методом -> 405 + заголовок `Allow`
- **Авто-HEAD**: HEAD откатывается к handler GET, если нет явного маршрута HEAD
- **Редирект завершающего слеша**: `/users/` <-> `/users` -> 301
- **Автокоррекция пути**: `//foo/../bar` -> `/bar` -> 301
- **Обнаружение конфликтов**: `/:id` + `/:name` на одном уровне -> ошибка при регистрации
- **Интроспекция маршрутов**: `ioh_router_walk()` для генерации документации, привязка к liboas
- **Метаданные маршрутов**: расширяемый `ioh_route_opts_t` с `oas_operation_t*` для liboas
- **Пользовательские handler**: настраиваемые handler для 404 и 405
- **Централизованная обработка ошибок**: handler возвращает int (0 или -errno) -> обработчик ошибок

### 5. Middleware (`src/middleware/`)

| Компонент | Строк | Описание |
|-----------|-------|----------|
| `ioh_cors.{h,c}` | ~100 | Предварительные запросы CORS + заголовки |
| `ioh_ratelimit.{h,c}` | ~200 | Token bucket по IP |
| `ioh_auth.{h,c}` | ~170 | Basic, Bearer, хуки JWT |
| `ioh_security.{h,c}` | ~220 | CSP, HSTS, X-Frame-Options, nosniff |
| `ioh_logging.{h,c}` | планируется | Структурированные JSON-логи доступа/ошибок |
| `ioh_metrics.{h,c}` | планируется | Lock-free thread-local метрики Prometheus |

**Архитектура метрик (lock-free):** Каждый рабочий поток поддерживает thread-local реестр метрик,
выровненный по 64-байтовым границам кеш-линий. Никаких атомарных операций на горячем пути
запроса -- счётчики инкрементируются локально. При сборе `/metrics` выделенный handler обходит
все thread-local блоки, агрегирует значения без блокировок и сериализует в текстовый формат
экспозиции Prometheus. Это устраняет конкуренцию за кеш-линии, которая снижает
производительность на 30-40% при использовании общих атомарных счётчиков.

**Стандартные метрики:**
- `ioh_http_requests_total{method,status}` -- счётчик
- `ioh_http_request_duration_seconds` -- гистограмма
- `ioh_http_connections_active{protocol}` -- gauge (h1/h2/h3)
- `ioh_tls_handshake_duration_seconds` -- гистограмма
- `io_uring_sqe_submitted_total` -- счётчик
- `ioh_bufpool_available` -- gauge

**Эндпоинты здоровья:** `/health` (живучесть, всегда 200), `/ready` (готовность: TLS-сертификаты,
пулы буферов, лимиты соединений).

### 8. Интеграция с liboas (`src/middleware/ioh_oas.c`)

liboas -- это **отдельный проект** -- библиотека OpenAPI 3.2.0 для C23. iohttp предоставляет
точки интеграции через middleware-адаптер, а НЕ встроенный движок OpenAPI.

| Компонент | Строк | Описание |
|-----------|-------|----------|
| `ioh_oas.{h,c}` | планируется | Middleware-адаптер liboas, хелперы монтирования/публикации |

**Архитектура интеграции (один поиск маршрута):**
1. При запуске liboas компилирует документ OpenAPI -> `oas_compiled_api_t`
2. Каждый маршрут iohttp хранит `oas_operation_t*` через `ioh_route_opts_t.oas_operation`
3. В рантайме router iohttp находит маршрут (один поиск) -> middleware получает уже найденную операцию
4. Middleware liboas валидирует запрос/ответ по схеме операции

**Что iohttp предоставляет для liboas:**
- Сопоставление `ioh_request_t` -> `oas_runtime_request_t`
- Сопоставление `ioh_response_t` -> `oas_runtime_response_t`
- `ioh_tls_peer_info_t` -> `oas_security_ctx_t` (метаданные TLS/mTLS)
- `ioh_conn_info_t` -> реальный IP клиента (после декодирования PROXY protocol)
- Хук валидации запроса перед handler
- Хук валидации ответа после handler
- Хелпер публикации `/openapi.json`

## Поток данных

### Запрос HTTP/1.1

```
1. io_uring multishot accept → new fd
2. (optional) Read PROXY protocol header → extract real client IP
3. wolfSSL_accept() via custom I/O callbacks
4. io_uring recv → provided buffer → wolfSSL_read() → plaintext
5. picohttpparser parse → ioh_request_t
6. Router lookup → middleware chain → handler
7. Handler writes ioh_response_t
8. Serialize headers → wolfSSL_write() → io_uring send
```

### Запрос HTTP/2

```
1-3. Same as HTTP/1.1 (ALPN selects h2)
4. io_uring recv → wolfSSL_read() → nghttp2_session_mem_recv()
5. nghttp2 callbacks fire per-stream → ioh_request_t per stream
6. Router → middleware → handler (per stream)
7. nghttp2_submit_response() → nghttp2_session_mem_send2()
8. wolfSSL_write() → io_uring send
```

### Запрос HTTP/3

```
1. io_uring recv (UDP) → ngtcp2_conn_read_pkt()
2. QUIC decryption via ngtcp2_crypto_wolfssl
3. nghttp3_conn_read_stream() → per-stream callbacks → ioh_request_t
4. Router → middleware → handler
5. nghttp3_conn_submit_response() → ngtcp2_conn_write_pkt()
6. io_uring send (UDP)
```

## Модель памяти

- **Пул соединений фиксированного размера** -- настраивается на этапе компиляции (по умолчанию 256)
- **Предоставляемые буферные кольца** -- управляемый ядром пул буферов для recv (без выделений
  на горячем пути). При `-ENOBUFS` (кольцо исчерпано) multishot recv перевооружается после
  пополнения буферов. Двойная буферизация: одна группа активна в ядре, вторая пополняется
  приложением.
- **Зарегистрированные буферы** -- `IORING_REGISTER_BUFFERS` закрепляет память для DMA, избегает обходов таблицы страниц
- **Зарегистрированные файлы** -- `IORING_REGISTER_FILES` предрегистрирует fd, пропускает поиск в таблице fd
- **Аллокаторы arena** -- время жизни на один запрос, освобождение после отправки ответа
- **Пути zero-copy** -- splice для статических файлов, `SEND_ZC` для больших ответов
- **`[[nodiscard]]`** -- на всех функциях выделения памяти
- **`static_assert`** -- проверки размеров структур на этапе компиляции (например, `sizeof(ioh_conn_t) <= 512`)
- **Отсутствие неконтролируемого роста кучи** -- все буферы ограничены, слишком большие запросы отклоняются

## Модель конфигурации

Двухуровневая конфигурация:

1. **Этап компиляции** -- `IOH_MAX_CONNECTIONS`, `IOH_MAX_HEADERS`, `IOH_ENABLE_HTTP2` и т.д.
2. **Этап выполнения** -- структура `ioh_server_config_t`, передаваемая в `ioh_server_create()`

```c
ioh_server_config_t cfg = {
    .listen_addr   = "0.0.0.0",
    .listen_port   = 8080,
    .tls_cert      = "/path/to/cert.pem",
    .tls_key       = "/path/to/key.pem",
    .max_connections = 256,
    .queue_depth   = 256,
    .keepalive_timeout_ms = 30000,
    .header_timeout_ms = 5000,
    .body_timeout_ms = 30000,
    .max_header_size = 8192,
    .max_body_size = 1048576,
    .proxy_protocol = false,
};
```

## Модель безопасности

### Усиление безопасности

**Защита от HTTP/2 Rapid Reset (CVE-2023-44487):** Принудительное применение
`SETTINGS_MAX_CONCURRENT_STREAMS` через настройки сессии nghttp2. Без ограничения потоков
сервер уязвим к атаке Rapid Reset (клиент открывает и немедленно сбрасывает потоки с высокой
частотой), которая вывела из строя продакшен-серверы по всей индустрии в октябре 2023.
Настройте разумный лимит (например, 100--256 одновременных потоков) и отслеживайте частоту
RST_STREAM -- если клиент отправляет RST быстрее порога (например, 100 RST/секунду),
разорвите соединение.

**Защита от Slowloris / Slow POST:**
- Таймаут чтения заголовков: 5--15 секунд через `io_uring_prep_link_timeout()`, связанный с
  начальным recv после accept. Если заголовки не получены полностью в пределах дедлайна,
  соединение закрывается.
- Принудительная минимальная скорость передачи: для тел запросов применяется минимальная
  скорость в байтах/секунду (например, 1 KiB/с). Клиенты, отправляющие данные медленнее,
  вероятно, являются зондами атак.
- Ограничение соединений по IP: отслеживание активных соединений по IP-адресу источника,
  отклонение новых соединений сверх лимита (например, 64 на IP). В сочетании с таймаутом
  заголовков это предотвращает исчерпание ресурсов атаками медленных соединений.

| Уровень | Защита |
|---------|--------|
| Сеть | Лимиты соединений, ограничение частоты запросов на accept |
| PROXY protocol | Белый список доверенных прокси, таймаут валидации заголовков |
| TLS | Усиленная конфигурация wolfSSL, ограничение наборов шифров, mTLS |
| HTTP-парсер | Макс. размер/количество заголовков, макс. длина URI, макс. размер тела |
| Маршрутизатор | Нормализация пути, защита от обхода директорий, отклонение NUL-байтов |
| Статические файлы | Изоляция корня документов, каноникализация, отключённый листинг директорий |
| Middleware | Ограничение частоты запросов, хуки аутентификации, CORS |
| Заголовки | CSP, HSTS, X-Frame-Options, nosniff, Referrer-Policy |
| Память | Ограниченные буферы, arena на запрос, explicit_bzero для секретов |
